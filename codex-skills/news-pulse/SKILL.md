# 企業ニュース速報：株価急変の要因を短時間で判定する

$ARGUMENTS に対して、直近ニュースの探索と株価変動の要因分析を行う。これは深い企業調査ではなく、**情報への迅速な初動対応**である。目的は短時間で次に答えることである。

- 対象企業で最近何が起きたか。
- 株価変動を最もよく説明する要因は何か。
- 投資仮説を再確認すべきか。

## 対象となる場面

- 保有銘柄または監視銘柄が大きく変動した。参考triggerは1日±5%、1週間±10%だが、銘柄の通常変動率も考慮する。
- 決算公表後の値動きについて、市場が何へ反応したかを確認したい。
- ニュースの見出しが、重要なsignalか一時的なnoiseか判断したい。
- **対象外**：完全な企業調査は`/investment-team`、決算精読は`/earnings-review`、長期の投資仮説追跡は`/thesis-tracker`を使う。

## 実行手順

### 手順1：対象と期間を確定する

$ARGUMENTS から次を抽出する。

| 項目 | 説明 | 既定値 |
|---|---|---|
| **会社名** | 日本語・中国語・英語の社名またはticker | 必須 |
| **調査期間** | 遡って調べる日数 | 14日。決算直後は7日を参考とする |
| **株価変動** | 上昇・下落率と期間。例：`3日で12%下落` | 任意。取得できる場合は実際の価格データで確認する |
| **重点** | 企業、規制、業界、市場心理 | 指定がなければ均等 |

会社だけが指定されている場合は、原則として直近14日を対象に通常点検を開始し、「具体的な株価変動がある場合は分析へ反映できる」と冒頭で示す。作業に不可欠でない確認質問だけを理由に止めない。

調査日と時間帯は`Asia/Tokyo`で記載し、海外市場の取引日・公表時刻は現地時間との違いを確認する。

### 手順2：情報入手可能性を評価する

`investment-team.md`の考え方を参考にするが、ニュース探索として次の区分を使う。

| 区分 | 特徴 | 探索方針 |
|---|---|---|
| **A（情報が多い）** | 大型株、報道量が多い、決算期 | 重複記事と見出しを除き、一次資料と実際の公表日時から原因を絞る |
| **B（情報が中程度）** | 中小型株、coverageが限定的 | 重要事象は原則として一次資料または2つの独立した情報源で確認する |
| **C（情報が少ない）** | 香港小型株、新規上場、注目度が低い | 該当ニュースが見つからない可能性も結論とし、需給・technical要因と未確認情報を分ける |

評価を全Agentへ伝える。

### 手順3：チームを作成する

TeamCreateを使う。

- `team_name`: `{会社名}-newspulse`。英小文字を使う。例：`pdd-newspulse`
- `agent_type`: `team-lead`

### 手順4：4つの探索Taskを作る

TaskCreateで次のTaskを作成する。

#### Task 1：企業イベント（company-event-scout）

- `subject`: `{会社名}の直近{N}日間の企業イベントを調べる`
- `description`：
  1. 取引所、SEC、HKEX、CNINFOなどの法定開示。
  2. 決算、業績予告、guidance、earnings call。
  3. 経営陣交代、役員・主要株主の売買、自社株買い、配当、株式報酬。
  4. 製品、公表済みの大型契約、M&A、事業売却。
  5. 増資、転換社債、ADR変更、重複上場、非公開化などの資本取引。
  6. 訴訟、行政処分、企業が開示したcompliance事案。
  7. 各項目へ**公表日時、事象発生日、一次資料、要約、株価変動との関連度（高・中・低）**を付ける。
  8. 公表日時の降順で時系列表を作る。

#### Task 2：規制・政策（regulatory-watcher）

- `subject`: `{業界または会社}に関わる直近{N}日間の規制・政策を調べる`
- `description`：
  1. 業界規制、制裁金、是正命令、license変更。
  2. 米中関係、関税、輸出管理、data security、上場規則。
  3. 付加価値税、法人税など企業利益へ関わる税制。
  4. antitrust、競争法、M&A審査。
  5. 医薬品調達、教育規制、不動産規制、platform規制など業界固有政策。
  6. 為替、金利、資本規制のうち対象企業への具体的経路があるもの。
  7. 各政策へ**公表日、施行日、情報源、影響度（直接・間接・関連薄い）**を付ける。
  8. 新たに確定した政策shockがあるかを判定する。

#### Task 3：業界・競合（industry-peer-analyst）

- `subject`: `{会社名}の業界と競合の直近{N}日間の動きを調べる`
- `description`：
  1. 主要競合3～5社の決算、製品、価格、経営陣の動き。
  2. 上流の原材料・supplierと、下流の顧客・channelにおける価格、能力、注文の変化。
  3. 出荷、需要、入札、利用量など業況指標。
  4. 代替技術・代替business model。
  5. 同業株および業界指数の値動き。対象企業が上回ったか、下回ったか、同方向か。
  6. **企業固有の事象か、業界betaか**を判断する。
  7. 各事象へ日付と情報源を付ける。

#### Task 4：市場心理と需給（sentiment-tracker）

- `subject`: `{会社名}の直近{N}日間の市場心理、institutional view、需給を調べる`
- `description`：
  1. 証券会社のrating、利益予想、目標価格の変更。予想変更日と根拠を確認する。
  2. 13F、Stock Connectなど、公表済みのinstitutional holdingと資金flow。開示の時間差を明記する。
  3. short interest、空売り比率、short report。
  4. 対象企業が段永平などの公開発言対象である場合に限り、次の補助ツールを使用できる。

```bash
python3 ~/ai-berkshire/tools/xueqiu_scraper.py \
  --user-id 1247347556 \
  --keywords {会社名},{ticker} \
  --output /tmp/dyp-{会社名}.md
```

  5. 未確認の噂、雪球（Xueqiu）、X、Redditなどの議論。事実の根拠にせず、拡散状況として扱う。
  6. 出来高、block trade、margin、support・resistanceなどtechnical要因。ただし因果を断定しない。
  7. **fundamental要因か、sentiment・flow要因か**を判断する。

### 手順5：4Agentを並行起動する

**同一メッセージ内でTaskツールを4回呼び出す。**

- `subagent_type`: `general-purpose`
- `run_in_background`: `true`
- `team_name`: `{会社名}-newspulse`
- `name`: `company-event-scout` / `regulatory-watcher` / `industry-peer-analyst` / `sentiment-tracker`

各Agentのprompt：

```text
あなたは{会社名}のニュース速報チームに所属する{役割名}である。{探索領域}について、直近{N}日間の事象を調べる。

調査期間：{開始日}～{終了日}
株価変動：{ユーザーが示した変動。なければ「通常点検」}
情報区分：{A/B/C}

Task #{Task番号}：{Task subject}

要件：
{Task description}

調査方法：
- WebSearchでは具体的な日付、正式名称、ticker、必要に応じて英語・中国語の検索語を使う。
- WebFetchで法定開示、決算資料、規制文書などの一次資料を読む。
- 公表日と事象発生日を区別する。
- 噂は独立した2情報源があっても確認済み事実とは扱わず、一次確認の有無を示す。
- 見出しと本文が一致しない場合は「見出しが誤解を招く」と記載する。
- 最終出力は自然な日本語とする。

出力：
1. 最重要の発見3～5点。
2. 公表日時の降順による時系列表。
   | 日時 | 事象 | 情報源 | 株価変動との関連度 | 影響の持続性 |
3. この領域が株価変動を説明できるか、確度とともに結論を出す。
4. 見つからなかった情報、矛盾、今後の開示を列挙する。
5. 確認済み事実、会社説明、市場の解釈、推測を分ける。

完了後：
1. TaskUpdateでTaskをcompletedにする。
2. SendMessageでtype: "message"、recipient: "team-lead"として完全な報告を送る。
```

### 手順6：進捗を共有する

- 報告を受け取るごとに、その領域の重要な発見を最大3点共有する。
- 4件すべてを受け取る。
- 受領後、SendMessageで各Agentへ`shutdown_request`を送る。

### 手順7：team-leadが要因を統合する

これは企業調査報告ではなく、**株価変動の要因判定報告**である。

#### 1. 一文の結論

30～60字を目安に、主因、次点、性質を先に示す。根拠が不足する場合は「確認できない」とする。

#### 2. 統合時系列

| 日時 | 領域 | 事象 | 情報源 | 要因としての重み |
|---|---|---|---|---|
| 2026-04-30 | 企業 | ... | ... | 🔴 高 |
| 2026-04-29 | 業界 | ... | ... | 🟡 中 |
| 2026-04-28 | 市場心理 | ... | ... | ⚪ 低 |

- 🔴 高：単独でも変動幅を相当程度説明し得る。
- 🟡 中：複数要因の一部として寄与する。
- ⚪ 低：背景またはnoiseである可能性が高い。

#### 3. 要因候補の比較

| 要因候補 | 支持証拠 | 反証・代替説明 | 確度 | 持続性 |
|---|---|---|---|---|
| 例：決算下振れ | 売上高が予想を5%下回り、marginも低下 | 一時要因という会社説明がある | 高 | 短期1～2週間の可能性 |
| 例：業界beta | 同業も同時に8%下落 | 対象企業の下落率は業界を大きく上回る | 中 | 業界環境次第 |

時系列の近さだけで因果としない。公表前から価格が動いた場合は、他の情報や需給要因も検討する。

#### 4. 性質を判定する

次のいずれかを選び、根拠を示す。

- [ ] **価値に関わる事象**：利益力、経済的な堀、経営陣、長期構造が変化し、投資仮説の再確認が必要である。
- [ ] **市場心理・technical要因**：確認できるfundamental変化がなく、flow、positioning、betaが中心である可能性が高い。
- [ ] **主因を確認できない**：変動幅に見合う公表事象が見つからない。未公表情報の存在を断定せず、情報漏れ、需給、見落としを可能性として区別する。
- [ ] **複合要因**：価値に関わる事象を市場心理や需給が増幅した。

#### 5. 領域別要約

4領域それぞれについて、3～5点と寄与度を示す。

#### 6. 次の対応

| 対応 | 推奨度 | 根拠 |
|---|---|---|
| `/thesis-tracker`で投資仮説を再確認 | | |
| `/earnings-review`で決算を精読 | | |
| `/management-deep-dive`で経営陣を再評価 | | |
| positionの見直しを検討 | | 条件と確認事項を示し、断定的な注文指示にしない |
| 追加情報を待つ | | |

#### 7. 今後7～30日の確認項目

- [ ] 公表予定の事象。
- [ ] 追跡する指標。
- [ ] 投資仮説を強化・毀損するsignal。

#### 8. 情報不足

解決できなかった疑問、見つからなかった一次資料、矛盾する情報、今後の開示待ちを明記する。推測で空欄を埋めない。

### 手順8：報告を保存する

`reports/{会社名}/{会社名}-news-{YYYYMMDD}.md`へ保存する。directoryがなければ作成する。

### 手順9：チームを削除する

TeamDeleteで資源を整理する。

## 重要原則

1. **網羅性より初動速度**：10～15分程度で要因候補を絞る。深掘りは対応するSkillへ渡す。
2. **列挙より要因判定**：ニュース件数を増やすのではなく、変動幅とtimingを説明できる事象を選ぶ。
3. **不明を認める**：主因を確認できないときは、そのまま記載する。未公表悪材料を事実として扱わない。
4. **保有の有無で結論を変えない**：証拠に基づき、都合の良い「一時的な下落」と決めつけない。
5. **同時発生と因果を分ける**：影響経路、規模、timingを確認する。
6. **情報量を尊重する**：C区分でニュースが見つからないこと自体を重要な結果として記録する。
7. **客観性**：出典を付け、事実、会社説明、市場解釈、推測を分ける。
8. **売買を代行しない**：要因と次の確認手順を示し、最終判断はユーザーが行う。
9. **本資料は調査・学習用であり、投資助言ではない。**
