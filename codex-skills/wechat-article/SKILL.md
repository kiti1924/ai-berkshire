# WeChat公式アカウント記事：執筆・編集・読者の3 Agentで仕上げる

$ARGUMENTS を深く調査し、そのまま公開工程へ進められるWeChat公式アカウント記事を作成する。3つのAgentが役割を分担する。Author Agentが初稿を書き、Editor Agentが構成と表現を整え、Reader Agentが対象読者の視点から理解可能性を確認する。

**対応する入力形式**：記事テーマ。例：`大規模言語モデルのOPD技術`、`Qwen3技術報告`、`バフェットがテクノロジー株を避けた理由`

---

## 設計思想

優れた記事は、次の3条件を同時に満たす必要がある。

1. **深さ**：読者が時間を使う価値のある事実、論理、独自の整理がある。Author Agentが担う。
2. **読みやすさ**：構成が明確で、情報密度と速度に無理がない。Editor Agentが担う。
3. **理解可能性**：想定読者が途中で置き去りにならない。Reader Agentが担う。

一人だけで書くと、書き手には明らかな論理が読者には伝わらないことがある。3 Agent方式は、外部視点を工程へ組み込むためのものである。

---

## Phase 1：調査と素材収集

### 手順1：記事の位置づけを確定する

$ARGUMENTS から次を抽出する。指定がない項目には既定値を使い、重大な曖昧さがない限り作業を止めない。

| 観点 | 確認内容 | 既定値 |
|---|---|---|
| **対象読者** | 技術知識、業界経験、記事を読む目的 | 技術的素養はあるが、当該分野の専門家ではない読者 |
| **深さ** | 入門 / 中級 / 専門的 | 中級。式を使う場合は意味を説明する |
| **長さ** | 目標文字数 | 3,000～4,000字 |
| **一次資料** | 論文PDF、図、原資料の取得要否 | 必要 |
| **文体** | formal / conversational / critical | 知的な友人へ説明するような対話的文体。最終出力は自然な日本語 |
| **掲載先** | WeChat公式アカウント、日本語媒体、社内記事など | WeChat公式アカウント向けの可読性を参考にする |

### 手順2：深く調査する

Agentツールで2～3件の調査Agentを**並行起動**する。

#### Research Agent A：中核内容

- 論文解説：論文PDF、supplement、著者資料を取得し、主要貢献、手法、図表、実験結果、限界を抽出する。
- 技術テーマ：一次論文、公式documentation、benchmark、最新の実装動向を調べる。
- 事業・投資テーマ：法定開示、公的統計、企業資料、業界資料から最新データと競争環境を調べる。

#### Research Agent B：背景と実利用

- 技術やテーマが実際にどこで使われているかを調べる。
- 採用企業、導入結果、失敗例を確認する。
- 歴史的な転換点と直近の動向を整理する。

#### Research Agent C（任意）：比較対象

- 競合する手法、製品、解釈を比較する。
- 歴史的な発展経路を整理する。
- 今後の方向について、確認済み事実、研究者の見解、独自推定を分ける。

中国語圏のテーマでは、正式名称や中国語検索語を使ってよい。ただし、中国語資料をそのまま機械翻訳せず、日本語読者に通じる論理と語順へ再構成する。

### 手順3：素材を整理する

調査Agentの結果から次を作る。

1. **中核命題**：記事が伝える一文。
2. **重要データ**：3～5点。出典、基準日、単位を付ける。
3. **図表一覧**：使用目的、出典、著作権・利用条件、保存先を記録する。
4. **記事outline**：6～8節の見出しと、それぞれの役割。
5. **反対証拠と限界**：記事の中核命題に都合の悪い事実も残す。

## Phase 2：Author Agentが初稿を書く

AgentツールでAuthor Agentを起動し、次のpromptを使う。

```text
あなたは深い技術・投資テーマを扱うAuthor Agentである。WeChat公式アカウントまたは同等の長文媒体へ掲載できる日本語記事を書く。

## 対象読者
{対象読者像}

## 文体
- 最終出力は自然な日本語とする。英語・中国語の語順を持ち込まない。
- 技術用語は初出で英語名または原語を併記し、その後は日本語または一般的な表記へ統一する。
- 学術論文の逐語訳ではなく、知的な友人へ説明するように書く。
- 比喩は理解を助ける場合だけ使い、一つの主軸を維持する。
- 式とデータは必要な範囲で示し、直後に意味を平易に説明する。
- emojiは使わない。
- mobileで読みやすい段落長を意識し、原則として一段落を4行程度以内にする。
- 事実、著者の主張、推定、評価を区別する。

## 素材
{整理済みの事実、データ、論点、出典、反対証拠}

## 構成
1. 冒頭3段落：重要なデータ、意外な事実、明確な問題提起のいずれかで始める。穏当な比喩や定型句から始めない。
2. 背景：なぜ重要か、何を解決するか。
3. 中核内容2～3節：技術的な深さを示し、各概念を平易に説明する。
4. 実証・事例：データと具体例で検証する。
5. 業界への意味と見通し：確認済みの影響と不確実な予測を分ける。
6. 結論：中核命題を簡潔に再提示する。誇張ではなく、単独で引用しても意味が通る文にする。

## 図
- 論文解説では、論文PDFの図を使用する前に利用条件を確認する。必要な原図を抽出し、`![説明](相対パス)`で実ファイルを参照する。`[図X：説明]`のようなplaceholderを残さない。
- 抽出時は`pdftoppm`で少なくとも900 DPIのPNGへ変換し、PILで対象図表を切り出す。文字が読めない場合は1,200または1,500 DPIへ上げる。
- 各画像は500KB以上を目安とし、軸、凡例、注記が読めることを実画像で確認する。容量だけで品質を判定しない。
- 図の出典、figure番号、必要なcaptionを明記する。内容を誤解させる切り抜きをしない。
- ファイルは`assets/{テーマ略称}/`に保存する。
- 論文以外の画像も、利用許諾または適切な引用条件を確認する。画像が不要なら無理に追加しない。

## 数式
- 行内数式は`$...$`、独立数式は`$$...$$`を使う。
- 純textで数式を代用しない。
- 数式の直後に、各記号と式の意味を日本語で説明する。

約{目標文字数}字の完全な初稿を作成する。
```

Author Agentの完了後、初稿ファイルが存在するか確認し、途中欠落、未置換placeholder、壊れた画像参照、出典不足がないか読む。

## Phase 3：Editor AgentとReader Agentが並行確認する

初稿完成後、**同一メッセージ**でEditor AgentとReader Agentを起動する。

### Editor Agent用prompt

```text
あなたは長文記事を担当するEditor Agentである。次の記事を、内容を薄めずに編集する。

## 確認基準
1. タイトル：内容を正確に表し、30字を大きく超えてmobile表示で切れないか。clickbaitになっていないか。
2. 冒頭：最初の3段落で問題と読む価値が伝わるか。
3. 構成：論理の飛躍、重複、前提の欠落がないか。
4. 深さと可読性：式や技術説明が、平易に見せるだけで説明不足になっていないか。
5. rhythm：段落・節が長過ぎず、後半で同じ説明を繰り返していないか。
6. 図：実ファイルが参照され、読者が必要とする位置にあるか。captionと出典が正しいか。
7. 結論：本文から導ける判断か。誇張や未検証の断定がないか。
8. 日本語：翻訳調、中国語語順、不自然な英語混在がないか。

## 記事全文
{完全な初稿}

## 出力
1. 全体評価を3～5文で示す。
2. タイトル案を2～3件示す。
3. 節ごとの修正案を、必要な箇所だけ「原文 → 修正案」で示す。
4. 最優先の改善点を3件示す。
5. 事実確認または出典追加が必要な箇所を列挙する。
```

### Reader Agent用prompt

```text
あなたは{対象読者像}に該当するReader Agentである。知識水準と実際の読書体験から記事を確認する。

## 読者像
{具体的な知識水準、関心、読み方}

## 記事全文
{完全な初稿}

## 答えること
1. 最初の3段落を読んで続きを読みたいか。理由は何か。
2. 理解できない、または読み直しが必要な文はどこか。
3. 式や技術説明を理解できたか。平易な説明は役立ったか。
4. 中核となる比喩は適切か。よりよい説明方法はあるか。
5. 長過ぎる、短過ぎる、または集中が切れる箇所はどこか。
6. 記事の中核命題を一文で説明できるか。
7. 他者へ共有する価値があるか。共有するとき何と説明するか。
8. 重要だが記事にない問いは何か。
9. 事実と筆者の評価を混同した箇所はないか。
```

## Phase 4：定稿する

### 手順1：2 Agentの指摘を統合する

| 問題 | Editor Agentの典型的な指摘 | Reader Agentの典型的な指摘 | 対応 |
|---|---|---|---|
| 冒頭が弱い | 問題提起が遅い | 最初の3段落で読む理由がない | 重要データ、意外な事実、対立する見方から書き直す |
| 技術説明が難しい | 式が密集している | 同じ段落を何度も読む | 式を減らす、図を使う、前提と直観を補う |
| paceが遅い | 一節が長い | 後半で集中が切れる | 重複説明を削り、節を分割・統合する |
| 結論が弱い | 本文の発見が収束しない | 共有理由が見つからない | 本文の証拠から導ける一文へ書き直す |
| 概念が飛ぶ | 接続がない | 突然理解できなくなる | 前提、定義、transitionを補う |

### 手順2：修正する

1. 両Agentが指摘した問題は原則として修正する。
2. Editor Agentだけの指摘も、内容の正確性、構成、重複に関わる場合は修正する。
3. Reader Agentだけの指摘は、対象読者全体に関わるかを判断して修正する。
4. 指摘が対立する場合は、正確性を損なわない範囲でReader Agentの理解可能性を優先する。
5. 指摘を反映する際に、根拠、例外、反対意見を削って結論だけを強くしない。

### 手順3：図を抽出・整備する

論文解説で図が必要な場合は、定稿前に実ファイルを作る。

1. **render**：対象pageを900 DPIからPNGへ変換する。画像が500KB未満、または文字が読めない場合は1,200または1,500 DPIへ上げる。

```bash
pdftoppm -png -r 900 -f {page} -l {page} paper.pdf /tmp/page
```

2. **位置確認**：150 DPI程度のpage全体画像でfigure位置と切り出し座標を確認する。
3. **crop**：PILでfigureと必要なcaptionを切り出し、必要に応じて`compress_level=1`で保存する。軸、legend、注記を欠落させない。
4. **保存**：`assets/{テーマ略称}/fig{番号}-{説明}.png`へ保存する。各画像は500KB以上を目安とし、実際の可読性も確認する。
5. **挿入**：`![説明](../../assets/{テーマ略称}/fig{番号}-{説明}.png)`で参照し、出典を記載する。

PDFの図を分析する際はpage画像を確認し、text抽出だけで図の内容を推測しない。

### 手順4：最終ファイルを作る

定稿をMarkdownへ保存し、末尾に原資料を示す。

```markdown
## 原資料

- 論文：{タイトル、著者、年}
- arXiv / DOI / 公式URL：{参照先}
```

投資・財務テーマでは、調査日、データ基準日、通貨、単位、会計期間を明記し、本資料が投資助言ではないことを記載する。

## 執筆上の禁止事項

1. **データを捏造しない**：出典を確認できない数値は削除するか、「推定」と根拠を示す。
2. **AI的な定型句を使わない**：「一緒に見てみよう」「注目すべきは」「言うまでもなく」など、内容のない導入を避ける。
3. **誇張しない**：「破壊的」「革命的」などは、比較可能な証拠がある場合に限る。
4. **数式を放置しない**：各式を日本語で説明する。
5. **数式はLaTeXで書く**：`$...$`または`$$...$$`を使う。
6. **図のplaceholderを残さない**：最終稿では実ファイルを参照する。利用条件と出典を確認する。
7. **表の注記を曖昧にしない**：概念、主体、方向を正確に書く。
8. **比喩を乱用しない**：主軸となる比喩を維持し、節ごとに別の比喩へ飛ばない。
9. **結論を本文から切り離さない**：共有しやすさのために事実を誇張しない。
10. **最終出力は日本語**：中国語・英語資料を利用しても、日本語として自然に再構成する。
