# 企業深掘りseries：8本の長文で一社を理解する

$ARGUMENTS について、WeChat公式アカウント、動画platform、その他の公開媒体へ掲載できる8本の長文seriesを作る。中核となる能力は単に「書くこと」ではなく、**事実、会計上の定義、series内の整合性を厳しく修正すること**である。

参考例：`reports/腾讯/《看懂腾讯》/`

---

## 1. 対象となる依頼

一社を教科書のように深く理解し、複数の独立した長文として公開する場合に使う。単発の企業調査との違いは次のとおりである。

- 8本、合計約12万字を目安に、認識の修正から意思決定frameworkまでを一つのseriesとして構成する。
- 各記事は単独でも読めるが、共通の財務数値、valuation、経営陣評価、価格条件を使う。
- 想定読者は、一社を理解するためにまとまった時間を使う意思がある人であり、証券会社向けの要約資料ではない。

**本Skillを使わない場面**：単発の企業調査は`/investment-research`、決算の速報・精読は`/earnings-review`、業界横断調査は`/industry-research`を使う。

## 2. 8本の構成template

| # | 記事template | 中核となる問い | 目安文字数 |
|---|---|---|---:|
| 01 | Xを理解したつもりでも、見落としていること | 認識を修正し、よくある3つの誤解を検証する | 4,000～5,000 |
| 02 | Xの経済的な堀 — `<事業の本質を一文で>` | 堀はどの程度強く、5年後・10年後も残るか | 6,000～8,000 |
| 03 | X最大の利益engine — `<最も収益力が高い事業>` | 中核事業は何か。収益力が続く理由は何か | 6,000～8,000 |
| 04 | Xの財務諸表に隠れた別の会社 — `<投資先・子会社・資産>` | 投資portfolio、子会社、未認識価値をどう扱うか | 8,000～10,000 |
| 05 | AIまたは現在の主要themeで、Xは勝者か敗者か | themeの影響を事業別に分解する | 8,000～10,000 |
| 06 | バフェットの観点でXの財務諸表を読む | 売上総利益率、FCF、ROE、SBC、資本配分を確認する | 8,000～10,000 |
| 07 | `<経営陣の公開発言>` — Xを経営陣へ託せるか | 資本配分、誠実性、successionを評価する | 8,000～10,000 |
| 08 | どの価格なら検討でき、何が起きれば仮説を撤回するか | 3scenario valuation、red line、position frameworkを示す | 10,000～12,000 |

`00-series-guide.md`を非公開の索引として追加する。企業特性によって独立した内容が足りない記事は、無理に水増しせず、他の記事へ統合するか本数を減らす。

## 3. 文体と構成

### 語調

- **結論と重要事実を先に置く**。冒頭は数値、検証済みの反直観的事実、明確な争点のいずれかとする。
- バフェット、マンガー、段永平（ドゥアン・ヨンピン）、李録（リー・ルー）の枠組みを使うが、名言の羅列にしない。
- 強気または弱気を先に決めず、事実、因果、結論の順で示す。
- 中核判断には、必ず反対証拠または代替説明を付ける。
- mobile previewで切り取られる冒頭18～20字だけでも、内容が伝わるようにする。
- 最終出力は自然な日本語の「だ・である調」とする。

### 避ける表現

| 避ける表現 | 問題 | 代替 |
|---|---|---|
| 明らかに / 必ず / 絶対 | 根拠以上に確実性を示す | データでは / 現時点の証拠では / このscenarioでは |
| 私は思う / 感じる | 根拠のない主観に見える | 本frameworkでは / 次の事実から |
| 教科書的 / 神業 | 内容を説明しない賞賛 | 具体的な判断、数値、結果を書く |
| 著しく過小評価 / 深刻な乖離 | 比較基準がない | 推定価値に対して何%のdiscountか示す |
| 完璧 / 欠点がない | 一面的である | 制約と反対証拠を併記する |

### title

- 反差のある検証済み数値、または市場の共通見解と異なる結論をhookにする。
- subtitleは中立的に内容を要約する。
- 「日本版X」「小さなバフェット」などの安易なlabelを避ける。
- 読者が認識できる企業名、事業名、財務用語を使う。
- clickbaitのために本文の確度を超える断定をしない。

## 4. 厳格なfact-check

### 執筆前に避ける偽の精密さ

1. **主観確率で加重した期待値**：`30% × A + 50% × B + 20% × C`のように、根拠の弱い確率を使って精密に見せない。scenario、trigger、方向、rangeを示し、主観確率を事実のように扱わない。
2. **第三者推計のMAU・share**：QuestMobile、七麦、CBNDataなどは定義により大きく異なる。最も信頼できる2情報源をanchorとして比較し、その他は補助的に使う。
3. **過去成長率の直線延長**：単年度の33%成長を5年間そのまま複利計算しない。高・基準・低scenarioと、成長鈍化の理由を示す。
4. **非上場企業の未開示持分**：ByteDanceやHaltiなど、公表されていない持株比率を一点推定しない。確認不能とし、根拠がある場合だけrangeを示す。
5. **単一原因への帰属**：競合失敗を一つの要因だけで説明しない。複数の原因、反証、確認不能な点を列挙する。
6. **通貨と会計期間の混同**：`亿元`を億円とせず、CNY、HKD、USDなどを付ける。年度、四半期、TTMを区別する。

### 修正時の7項目

```text
□ 1. series内の数値整合：時価総額、Non-IFRS利益、持株比率などを全記事でそろえる
□ 2. 会計上の定義：IFRS / GAAP / Non-IFRS / SBC調整後 / FCFを明示する
□ 3. 二重計上：連結済み子会社を投資portfolioへ再計上せず、SOTPで重複させない
□ 4. 比較可能性：本体PERだけnet cashや投資資産を除き、競合は除かないような不公平な比較をしない
□ 5. 根拠の弱い確率加重を削除する
□ 6. 絶対的・誇張表現を検索して弱める
□ 7. 財務諸表以外の重要データへ情報源を付ける
```

絶対表現の確認例：

```bash
grep -R -nE "明らか|必ず|絶対|完璧|教科書的|著しく過小評価" reports/{会社名}/
```

### 企業固有の既知riskを先に記録する

執筆前に、当該企業で誤りやすい論点を一覧化する。例：

- historical returnは、最初の投資額だけでなく追加投資を含む累積投入額で計算する。
- 持株比率は最新の法定開示と株式数を使う。過去の比率を現在へ持ち込まない。
- 現物配当・分配の会計処理は、IFRIC 17その他の適用と認識日を確認する。
- 自社株買いがあっても、SBCの付与時期によって発行済株式数が一時的に増える場合がある。
- ADR比率、株式分割、dual-class voting rightsを確認する。

特定企業の古い数値を普遍的な規則として埋め込まず、執筆時点の資料で更新する。

## 5. 実行工程

### Phase 1：調査

記事01・02を書く前に次を行う。

1. 直近5年の年次報告書と最新の四半期・半期報告書を読む。
2. 少なくとも3件の独立した証券会社または調査機関の資料を読み、共通見解と反対見解を分ける。
3. `/investment-team`または`/investment-research`で内部用の調査底稿を作る。
4. 8本の中核命題を一覧化し、必要な場合はユーザーの明示条件と照合する。
5. 重要データは`skills/financial-data.md`に従ってcross-checkする。

### Phase 2：執筆

01から08の順で書く。前の記事で導入した概念と数値を引き継ぐ。

- 各記事を`reports/{会社名}/《{会社名}を理解する》/0X-{slug}.md`へ保存する。
- ユーザーが明示的に依頼するまでcommit、push、公開を行わない。
- 修正依頼を受けたらfact-check後に修正する。
- 変更履歴と未解決事項を記録する。

既存seriesが中国語名のdirectoryを使う場合は、後方互換のため勝手に改名しない。

### Phase 3：series横断の整合確認

8本の完成後、Explore Agentを並行起動して次を確認する。

1. 同じ数値が記事間で一致するか。
2. FBS、SBC、Non-IFRSなどの用語が初出で定義されているか。
3. 「記事06で詳述」のような参照先が実在し、内容も一致するか。
4. 各記事の要点と本文の数値が一致するか。
5. 事実の基準日が異なる場合、その違いを説明しているか。
6. 同じ主張が複数記事で不要に繰り返されていないか。

### Phase 4：公開前確認

repository固有のprivacy文字列、個人path、未公開情報を検索する。固定された個人名や過去のmachine pathを正本へ埋め込まない。

```bash
grep -R -nE "(/Users/|D:\\\\Documents\\\\|<内部名>|<未公開会社名>)" reports/ | head
```

実際のprivacy対象語はユーザーまたはrepositoryのprivacy規則から取得する。検査後も、ユーザーから明示的に依頼されるまでcommit、push、Pull Requestを行わない。

## 6. 修正依頼への対応

### 1. 先に事実を確認する

ユーザーから数値誤りを指摘された場合、直ちに文言だけを置換せず、次を比較する。

- ユーザーが示した数値と情報源。
- 法定開示または企業公式資料。
- repository内の既存earnings・調査報告。
- 自分が使用した数値、期間、通貨、会計上の定義。

「ユーザー提示値 / 一次資料 / 既存記載」の3者を示し、正しい定義を決める。

### 2. 修正level

| level | 種類 | 対応 |
|---|---|---|
| 🔥 hard error | 数値、因果、通貨、会計定義が誤っている | 必ず修正する |
| ⚠️ 主観過多 | 強い断定、誇張、clickbait | 弱めるか削除する |
| 🔬 granularity | 出典、定義、補足の不足 | 可読性と重要性に応じて追加する |
| ❓ 信頼性不足 | 第三者推計の差が大きく解決できない | 無理に平均せず、削除またはrange・不明とする |

### 3. 連動箇所を確認する

一つの数値を直すときは、全seriesで派生値と参照箇所を検索する。

- 時価総額変更：PER、事業価値、discount、FCF Yield。
- 持株比率変更：資産ranking、historical table、売却履歴。
- 用語変更：初出定義、本文、要点、相互参照。
- 発行済株式数変更：1株当たり価値、SBC、希薄化。

### 4. commit・push後の報告

commitまたはpushはユーザーが明示的に依頼した場合だけ行う。実施した場合は次を報告する。

```text
反映状態：{commitのみ / push済み}
commit：{SHA}
修正件数：{N}
- 修正内容
- 連動修正
- 未修正・未解決
```

## 7. 本Skillが行わないこと

- 読者の個別事情を知らずに売買判断を代行しない。各記事に「調査・学習用であり、投資助言ではない」と記載する。
- 短期株価を予測しない。scenario、trigger、rangeを示す。
- 主観確率による加重期待returnを計算しない。
- 著名投資家が保有していることだけを投資根拠にしない。
- 独立した内容がないのに8本へ水増ししない。
- 実在人物の架空の発言を直接引用として作らない。

---

## 8. complianceとprivacy

- 公開報告では、法定開示、企業公式資料、公的統計、公開された調査資料など、公開情報だけを使う。
- ユーザーの勤務先内部名、internal message、非公開保有、認証情報、local machine固有情報を含めない。
- 公開前にrepositoryのprivacy規則へ従って検索する。
- 著者名や所属は、ユーザーが指定した公開identityだけを使い、異なるidentityを混在させない。
- 引用・画像・図表は、出典と利用条件を確認する。

**このseriesの品質は、文章の派手さではなく、修正の厳密さで決まる。** 偽の精密さ、主観確率、会計定義の混同、二重計上、絶対表現を、執筆前と完成後の両方で除去する。
